# 🐈 MeowJob 貓咪打工平台：需求規格與架構設計決策手冊

本手冊詳實記錄了「MeowJob 貓咪打工平台」在開發過程中，針對**需求規格書的審查討論、架構決策與邊界防呆設計**的歷程。這是一份將技術實作與業務邏輯緊密結合的協作紀錄，旨在說明「我們為什麼要這樣設計網頁」。

---

## 💡 核心哲學：高隱私性與零維護成本的「自助式網頁」

在討論之初，我們為平台定調了三大基礎架構：
1.  **純前端無後端（Serverless）**：不架設收集個資的雲端資料庫，貓咪生活與就醫個資 **100% 儲存在使用者瀏覽器的本地保險箱（LocalStorage）**。對外分享則將資料加密壓縮後塞在網址尾巴（URL）傳遞，確保隱私絕不外洩。
2.  **Vue 3 全域腳本版（CDN）**：首頁使用 Vite 模組化編譯，但本貓咪平台（`cats/`）為了讓使用者能在電腦資料夾**直接雙擊 HTML 檔案離線點開即用**，放棄了會被瀏覽器安全政策（CORS）阻擋的 `type="module"` 載入法，改採全域腳本引入，達成離線與線上 100% 雙向相容。
3.  **GitHub Pages 靜態託管**：利用 GitHub 免費且高穩定的靜態網頁服務與 GitHub Actions 自動化上架工廠（CI/CD），省去所有主機租用開銷，並自動綁定免費的 SSL 安全加密連線（HTTPS）。

## 🛡️ 第一部分：安全性與隱私防護討論

我們針對規格書中可能引發的安全漏洞與隱私外洩進行了深入探討，並實作了以下防護：

### 1. 轉帳金流的隱私風險防護
*   **問題**：規格書中原本包含銀行帳號等明細，但因為資料會被打包在分享網址中公開傳遞，任何拿到連結的人都能輕易還原出飼主的銀行帳號。
*   **決策**：在填寫介面新增**顯眼的隱私警示警語**，建議並引導使用者優先使用二維條碼（QR Code）圖片而非填寫完整銀行帳號文字，保障個資安全。

### 2. 惡意代碼注入防護（XSS 攻擊防禦）
*   **問題**：訪客開啟的貓咪卡片資料完全來自網址解碼後的內容。若有惡意使用者修改網址參數，在貓咪姓名或日誌中塞入惡意腳本（如 `<script>`），網頁可能會被劫持。
*   **決策**：
    *   全站文字渲染一律使用 Vue 的安全文字插值（自動進行 HTML 字元轉義防禦），**嚴禁使用 `v-html`**。
    *   針對大頭貼與日誌圖片網址，嚴格限制協議開頭必須為 `http://`、`https://` 或 `data:image/`，徹底防範以 `javascript:` 開頭的惡意注入。

### 3. 本地儲存容量限制與防當機機制
*   **問題**：瀏覽器的本地保險箱（LocalStorage）只有 5MB 共享額度。若使用者上傳過多大型圖片，會觸發容量爆滿錯誤，導致網頁直接當機或變白畫面。
*   **決策**：
    *   **限制上傳體積**：限制本機上傳的大頭貼、貼文圖片、QR Code 單一檔案大小**不得超過 300 KB**。
    *   **容量指示條**：在大廳最上方提供動態進度條，顯示目前已使用幾 MB（如「已使用 1.2 / 5MB」）。
    *   **當機捕捉**：以 `try...catch` 嚴密包覆寫入動作，當遇到 `QuotaExceededError`（容量爆滿）時，攔截錯誤並彈出友善提示：「*儲存空間已滿，請清理圖片或改用外部圖床網址。*」

## ⚙️ 第二部分：資料邏輯與邊界防呆設計

針對一些特殊的邊界情況，我們在需求審查中定義了清晰的處理規則，以確保極致的使用者體驗：

### 1. 體重折線圖的邊界情況（資料不足時）
*   **問題**：SVG 趨勢折線圖至少需要 2 筆資料才能拉出線條。如果貓咪只有 0 筆或 1 筆體重紀錄，網頁若強行畫線會崩潰。
*   **決策**：
    *   **0 筆資料**：圖表區塊顯示提示文字「*尚無足夠資料繪製趨勢*」。
    *   **1 筆資料**：在 SVG 畫布正中央繪製單一圓點標記，不繪製折線。
    *   **2 筆（含）以上**：正常計算圓滑貝氏曲線與草本綠漸層。

### 2. 雙排序分離邏輯
*   **問題**：使用者要求「體重明細清單必須按日期從最新到最舊（降冪）排列」，但 SVG 折線圖畫線必須「從左到右按時間由舊到新（升冪）排列」，兩者排序邏輯互相衝突。
*   **決策**：在 Vue 中建立兩個不同的計算屬性。`sortedWeightHistory` 負責維持時間升冪，供 SVG 繪圖引擎拉線；`displayWeightHistory` 負責日期降冪，供前端歷史清單渲染。兩者分離，完美解決圖表軌跡錯亂與列表排序的需求。

### 3. 刪除分類後的「孤兒資料」防呆
*   **問題**：若飼主自訂了醫療類別（例如「中醫調理」）並新增了醫療紀錄，隨後到設定中刪除該類別，會導致該醫療紀錄失去關聯而報錯。
*   **決策**：採用防呆機制——不強制轉換或阻擋刪除。若醫療紀錄引用的分類已不存在，系統會自動在分類欄位保留原文字，並在後方後綴加註「**(已刪除分類)**」，防止程式崩潰。

### 4. 網址解析損毀處理（優雅降級）
*   **問題**：若分享網址在社群軟體傳送過程中被部分截斷或損毀，解碼器（Decompress）會報錯並呈現白畫面。
*   **決策**：在網址解析的最外層加入 try-catch 攔截。一旦解碼失敗，將網頁引導至特製的 error 錯誤頁面，呈現療癒插圖「*喵！連結好像被貓咪叼走了……*」，並提供「返回大廳」的導流按鈕。

## 📈 第三部分：功能優化與介面微調討論

隨著開發進行，我們針對實際操作體驗進行了多次微調，這些是原規格書未提及但成品已完美實現的細節：

### 1. 體重折線圖橫軸的視覺重構
*   **問題**：原本 X 軸日期貼在資料點下方容易重疊，且數值標示不夠清晰。
*   **決策**：
    *   **粗橫軸線**：在 Y 軸座標 `210` 繪製一條粗度為 `3` 的深色基底線。
    *   **日期置上**：將 `yyyy/mm/dd` 的日期文字擺放在粗線的**正上方**（Y 座標 `198`），使其不被邊界裁剪，且閱讀極為整齊。
    *   **數值加大**：將圓點上方的體重文字級數**加大兩級**（大小設為 11），方便在手機螢幕上清晰閱讀。

### 2. 體重清單的點擊展收與刪除
*   **問題**：體重明細若一直顯示會佔用過多空間。
*   **決策**：設計「點擊圖表切換展收」的互動。使用者只要點擊折線圖的任何地方，下方即會展開降冪體重列表，且非訪客模式下每一筆資料後方都提供「刪除」按鈕，能隨時修正體重並即時重新渲染折線圖。

### 3. 日誌與醫療紀錄的線上編輯/刪除
*   **問題**：原需求只要求發布，未提及後續修改與刪除。
*   **決策**：在大廳非訪客模式下，時間軸貼文與就醫紀錄旁均新增「編輯 ✏️」與「刪除 🗑️」按鈕。點選編輯會自動回填表單供修改儲存；點選刪除會跳出確認視窗，確認後自本地保險箱徹底移除。

### 4. 履歷工作經歷自訂編輯器
*   **問題**：貓咪的打工履歷需要能夠隨時增刪不同的打工職稱與描述。
*   **決策**：在編輯貓咪資料的 Modal 表單底部，新增「工作經歷編輯區」。使用者可以動態點選「➕ 新增工作經歷」，表單會像卡片一樣展開，可填寫「職稱、期間、描述」，並能對個別經歷進行卡片刪除，儲存後立即更新履歷頁籤。
