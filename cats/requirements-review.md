# 🔍 MeowJob 需求規格書審查與修改建議

*(本檔案為 `requirements.md` 的全端工程師審查意見，非正式規格書本體，僅供開發前討論與後續修訂參考。)*

---

## ✅ 第一輪審查（已於 requirements.md 修改完成）

以下 12 項已依建議修改進 `requirements.md`，此處保留原始審查意見作紀錄。

### 🔴 高風險項目

**1. 「串接實際的支付管道」與「純前端無後端」自相矛盾**
真正的金流串接（LINE Pay、綠界 ECPay、街口等）都需要後端做簽章、Callback 驗證，純靜態網頁不可能做到。
✅ **已修改**：一、4 改寫為「展示收款資訊」，並註明金流自動化串接列為未來版本規劃、本期不實作。

**2. Base64 分享連結會把銀行帳號整組明碼公開**
`bankAccount`、`paymentQrCode` 沒有被排除在分享過濾邏輯外，連結被轉傳出去任何人都能解碼看到真實帳號。
✅ **已修改**：三、1 收款與基金欄位下新增隱私提醒警語。

**3. 分享連結是攻擊者可控輸入，卻沒有 XSS / 資料驗證規範**
`blogPosts`、`personalityTags`、`medicalRecords` 等欄位來自 URL 參數解碼後的 JSON，等於外部可控輸入，原規格只處理解碼失敗，沒處理渲染安全。
✅ **已修改**：新增四、6「分享資料結構驗證與渲染安全 (Schema Whitelist & Rendering Security)」章節。

**4. Base64 URL 長度機制只是「警告」，沒有真正壓縮**
純 Base64 會讓資料膨脹約 33%，完全沒利用瀏覽器內建壓縮 API。
✅ **已修改**：四、2 新增「編碼前先行壓縮」，改用 `CompressionStream('deflate-raw')` + Base64URL；二、2 流程說明同步更新。

### 🟡 規格矛盾與遺漏

**5. 分享連結無法產生正確的社群預覽卡片（OG Meta）**
純前端 CSR 架構下，分享連結被貼到 LINE/Messenger/Discord 時只能顯示首頁固定 OG 標籤。
✅ **已修改**：二、4「已知限制與後續規劃」新增此限制說明，列為未來版本規劃。

**6. 測試項目 9 與第四章 1 的 300KB 限制範圍不一致**
第四章原文只寫「QR Code 圖片」限制 300KB，但驗收測試第 9 項卻是測日誌貼文圖片。
✅ **已修改**：四、1 統一為「所有本機上傳圖片（大頭貼、日誌照片、QR Code）皆適用 300KB 限制」。

**7. 缺少「刪除貓咪」與「醫療類別刪除後的孤兒資料」處理**
✅ **已修改**：三、1 新增「貓咪刪除」流程（需二次確認）；三、2 新增類別刪除防呆說明。

**8. LocalStorage 5MB 配額是「整個網域共用」，不是「單隻貓」**
✅ **已修改**：四、1 新增「儲存空間指示」條與 `QuotaExceededError` 攔截說明。

**9. 缺少 Schema 版本號**
✅ **已修改**：三、1 新增 `schemaVersion` 欄位。

### 🟢 較次要的建議

**10. 貓年齡換算公式未定義**
✅ **已修改**：三、1 補上明確換算公式（第一年 15 歲、第二年 +9、之後每年 +4）。

**11. 字體效能風險**
✅ **已修改**：五、2 新增 `font-display: swap` 與僅載入必要字重的建議。

**12. 多分頁同步問題**
✅ **已修改**：二、4「已知限制與後續規劃」納入此限制，註記本期不處理。

## 🆕 第二輪審查（尚未寫入 requirements.md，待確認優先順序）

### 🔴 高風險項目

**13. 罐罐進度 (`currentCansCount`) 如何更新，完全未定義**
**對應原文**：三、1（第 61 行）。

因為沒有後端可以驗證真實付款，系統不可能知道訪客是否真的轉帳了。目前規格沒說 `currentCansCount` 是飼主自己手動輸入更新（自報進度），還是訪客點擊「餵顆乾乾」時前端自動 +1（純粹好玩、與真實贊助金額無關）。兩種語意差很多，會讓「月薪罐罐目標」進度條的意義混亂，也可能誤導訪客以為點免費按鈕等於贊助生效。

**建議修改**：明確定義 `currentCansCount` 為「飼主自行手動更新的自報進度數字」，與「餵顆乾乾」的 Canvas 特效互動完全脫鉤（特效純粹是免費的趣味互動，不寫入任何持久化數字）。

**14. 新增的 `CompressionStream` 壓縮方案在舊版 Safari 不支援**
**對應原文**：四、2（第 88 行）vs 六、3 相容性指標（第 169 行）。

`CompressionStream` 是較新的瀏覽器 API，舊版 Safari 或部分 iOS WebKit 版本可能不支援，但驗收標準明確要求 Safari 需正常運作。

**建議修改**：新增 feature detection，偵測不到 `CompressionStream` 時自動降級為純 Base64 編碼（犧牲連結長度換取相容性），並在規格書中明確寫出降級邏輯。

**15. 前端路由機制完全沒提到，可能導致 GitHub Pages 重新整理 404**
規格書全篇沒說明「大廳 → 貓咪詳細頁 → 履歷/日誌/醫療/打賞頁籤」之間怎麼切換。若實作時使用瀏覽器 History API（例如 `/cats/mikan/resume` 這種路徑），因部署方式是純靜態檔案 `cp -r cats dist/`、沒有 SPA fallback 規則，訪客重新整理子頁面會直接 404。

**建議修改**：在規格書中明確要求站內導覽一律用前端狀態切換（`v-if`/`v-show`）或 hash（`#/...`），不使用 History API pushState 路由，僅有最外層的 `?share=` 查詢字串允許出現在網址上。

### 🟡 中等缺口

**16. 備份匯出/匯入的範圍未定義**
**對應原文**：六、1 驗收表「備份匯出」列（第 159 行）、七、測試項目 12（第 235-238 行）。

只提到「還原大廳的所有貓咪列表」，沒說是否連同 `meowjob_settings`（自訂醫療類別）一起匯出/匯入。若只匯出 `meowjob_cats`，還原後醫療紀錄可能引用到不存在的自訂類別。

**建議修改**：明確定義備份檔案需同時包含 `meowjob_cats` 與 `meowjob_settings` 兩份資料。

**17. 匯入備份時的 ID 衝突處理未定義**
若使用者在已有貓咪資料的瀏覽器中匯入一份舊備份，兩邊的 `id`（時間戳記）可能重複，規格沒定義匯入策略是「整批覆蓋」還是「合併新增」，也沒定義 ID 衝突時怎麼處理。

**建議修改**：明確定義匯入行為（建議：整批覆蓋現有大廳資料，匯入前跳出確認彈窗告知「將覆蓋目前所有貓咪資料」）。

**18. 體重折線圖資料筆數過少時的顯示未定義**
只有 0 筆或 1 筆體重紀錄時無法畫出「線」，規格沒定義這種邊界情況要顯示什麼。

**建議修改**：新增說明——少於 2 筆資料時顯示單點標記或提示文字「尚無足夠資料繪製趨勢」，不強行渲染折線。

**19. 無障礙 (Accessibility) 完全沒有規範**
整份規格書沒有任何 a11y 要求，包括自訂日曆組件的鍵盤操作、色彩對比（例如 `#E76F51` 橘色文字在 `#FFFDF9` 米白背景上的對比度是否符合 WCAG AA）、Canvas 特效需標記為裝飾性內容（`aria-hidden`）、圖片替代文字。

**建議修改**：在五、UI/UX 章節新增基本 a11y 檢查項：色彩對比需符合 WCAG AA、互動元件需可鍵盤操作、裝飾性動畫加 `aria-hidden="true"`、圖片需有 `alt` 文字。

### 🟢 較次要的建議

**20. 圖示庫（Lucide Icons）載入方式未定義**
CDN 引入還是本地打包會影響 1.5 秒的 DOMContentLoaded 效能指標，但規格沒指定。

**建議修改**：明確指定載入方式（建議本地打包或僅引入所需圖示子集，避免額外的 CDN 網路往返時間）。

**21. 「公開健康摘要」用詞與實際 Schema 不完全對應**
**對應原文**：一、3（第 11 行）。

目標描述提到「公開健康摘要」，但三、Schema 只有逐筆 `medicalRecords.isPublic` 過濾，沒有「摘要/聚合」層級的欄位（例如最新體重、健康狀態燈號）。

**建議修改**：若只是逐筆列表也可以，建議把措辭改成更精確的「公開醫療紀錄清單」以免預期落差；若真的要做聚合摘要則需額外定義欄位。

**22. 第七章測試核對表未涵蓋新增的防呆項目**
第一輪修改新增的防呆設計（分享連結注入防護、儲存空間配額提示、刪除貓咪、刪除醫療類別孤兒資料）都還沒有對應的自我測試項目。

**建議修改**：於七、自我功能測試核對表新增對應測試項目：
- 惡意竄改分享連結注入 script 標籤測試（驗證不會執行、不會 XSS）。
- 儲存空間超過配額測試（驗證 `QuotaExceededError` 攔截與提示）。
- 刪除貓咪測試（含二次確認彈窗）。
- 刪除醫療類別後，既有紀錄顯示「(已刪除分類)」測試。
